Guía de Aprendizaje BBDD Avanzada

Proyecto: Sistema de Gestión TECDA Maniquí | Nivel: Intermedio/Avanzado

1. Modificación Estructural (DDL) actualizacion_esquema.sql

¿Por qué?

Los requerimientos del negocio cambian (ej. agregar costos). La base de datos debe evolucionar sin perder la información ya existente.

¿Cómo?

Usando comandos `ALTER TABLE` para agregar columnas, `RENAME` para renombrar tablas y `CHANGE` para modificar tipos de datos.

¿Cuándo?

Durante actualizaciones de versión del software o cuando surgen nuevas necesidades comerciales (ej. control de márgenes de ganancia).

¿Dónde?

Directamente sobre el esquema lógico de la base de datos.

¿Para qué?

Para adaptar el sistema a la realidad sin tener que borrar y recrear la base de datos desde cero.

🧠 Qué se aprende:

Manejo de lenguaje de definición de datos (DDL), alteración segura de esquemas en producción y diseño incremental.

2. Triggers (Disparadores) trigger_serial_inteligente.sql

¿Por qué?

Para garantizar que ciertas reglas de negocio (como el formato de un número de serie) se cumplan siempre, sin depender del código de la aplicación.

¿Cómo?

Interceptando el evento `INSERT` justo antes de que ocurra (`BEFORE INSERT`), calculando el nuevo valor y asignándolo a `NEW.columna`.

¿Cuándo?

Cada vez que se intenta guardar un nuevo registro en la tabla monitoreada.

¿Dónde?

En el motor de la base de datos, actuando como un "guardián" invisible de la tabla.

¿Para qué?

Para automatizar la creación de la "Nomenclatura Inteligente" y evitar errores humanos al teclear códigos.

🧠 Qué se aprende:

Lógica procedural en SQL, variables temporales (`DECLARE`), concatenación compleja, eventos DML y la integridad referencial automatizada.

3. Vistas (Views) vista_produccion_maniquies.sql

¿Por qué?

Las consultas analíticas suelen ser muy largas y complejas (con muchos JOINs). Repetirlas es propenso a errores y difícil de mantener.

¿Cómo?

Guardando la consulta bajo un nombre con `CREATE VIEW`, creando una "tabla virtual".

¿Cuándo?

Cuando necesitas generar reportes, dashboards o dar acceso a datos resumidos a otros usuarios.

¿Dónde?

En la capa de presentación de la base de datos, encapsulando la complejidad de las tablas subyacentes.

¿Para qué?

Para tener un Panel de Control que calcule en tiempo real el costo total de producción y ganancias sin almacenar datos redundantes.

🧠 Qué se aprende:

Abstracción de datos, seguridad por ocultamiento de esquema, uso avanzado de `JOIN` y funciones de agregación (`SUM`, `GROUP BY`).

4. Procedimientos Almacenados 01_procedimientos_almacenados.sql

¿Por qué?

Para agrupar múltiples pasos lógicos en una sola llamada, reduciendo el tráfico de red entre la aplicación y la base de datos.

¿Cómo?

Usando `CREATE PROCEDURE` para definir un bloque de código SQL que acepta parámetros de entrada (`IN`).

¿Cuándo?

Al ejecutar procesos que involucran varias tablas a la vez, como el "Ensamblaje de un Maniquí".

¿Dónde?

Se ejecutan directamente en el servidor de base de datos (Backend profundo).

¿Para qué?

Para centralizar la lógica de negocio. Si cambia la forma de ensamblar, solo cambias el procedimiento, no el código del sistema.

🧠 Qué se aprende:

Encapsulamiento, paso de parámetros, uso de `LAST_INSERT_ID()`, y optimización del rendimiento cliente-servidor.

5. Transacciones (ACID) 02_transacciones.sql

¿Por qué?

Para proteger la consistencia de los datos. No quieres registrar una venta si el pago falló en el último milisegundo.

¿Cómo?

Envolviendo las consultas entre `START TRANSACTION` y `COMMIT`. Usando `ROLLBACK` si algo sale mal.

¿Cuándo?

En operaciones críticas que modifican varias tablas de forma interdependiente (ventas, transferencias, control de stock).

¿Dónde?

En el control de concurrencia del motor transaccional (InnoDB en este caso).

¿Para qué?

Para garantizar que la base de datos "o hace todo el trabajo, o no hace nada", evitando datos corruptos o a medias.

🧠 Qué se aprende:

Principios ACID (Atomicidad, Consistencia, Aislamiento, Durabilidad), manejo de excepciones (`SQLEXCEPTION`) y seguridad de datos.

6. Índices 03_indices.sql

¿Por qué?

Leer una tabla de millones de registros fila por fila es ineficiente y lento.

¿Cómo?

Creando estructuras adicionales (`CREATE INDEX`) que actúan como el índice de un libro.

¿Cuándo?

Cuando notas que ciertas consultas (especialmente con `WHERE`, `JOIN` u `ORDER BY`) se vuelven lentas al crecer los datos.

¿Dónde?

Son estructuras físicas guardadas en el disco junto a los datos de la tabla.

¿Para qué?

Para que la búsqueda de un maniquí por su estado o de una pieza por su serial sea matemáticamente instantánea (complejidad O(log n)).

🧠 Qué se aprende:

Estructuras de datos en árboles B (B-Trees), optimización de lecturas vs. penalización en escrituras, y diseño de rendimiento.

7. Funciones (UDF) 04_funciones.sql

¿Por qué?

Para evitar repetir fórmulas matemáticas o transformaciones de texto en cada consulta que escribes.

¿Cómo?

Creando un bloque `CREATE FUNCTION` que acepta parámetros y obligatoriamente `RETURNS` (devuelve) un valor.

¿Cuándo?

Cuando necesitas aplicar cálculos específicos (ej. descuentos, impuestos) fila por fila en un `SELECT`.

¿Dónde?

Se integran directamente en el lenguaje de consultas del usuario.

¿Para qué?

Para poder hacer `SELECT CalcularDescuento(precio) FROM Modelos`, manteniendo las consultas limpias y estandarizadas.

🧠 Qué se aprende:

Extensibilidad de SQL, diferencias entre Funciones (devuelven valor) y Procedimientos (ejecutan acciones), y reutilización de código.

8. Permisos y Usuarios 05_permisos_usuarios.sql

¿Por qué?

Por seguridad. El usuario que conecta tu página web no debería tener poder para borrar la base de datos completa.

¿Cómo?

Creando usuarios (`CREATE USER`) y asignando privilegios precisos con `GRANT` o quitándolos con `REVOKE`.

¿Cuándo?

Antes de poner el sistema en producción o al conectar diferentes aplicaciones (ej. App de Ventas vs. App de Recursos Humanos).

¿Dónde?

En el diccionario de datos del motor de MariaDB, a nivel global o a nivel de base de datos específica.

¿Para qué?

Para implementar el Principio de Menor Privilegio: dar solo el acceso necesario para hacer el trabajo (ej. solo `SELECT` a los vendedores).

🧠 Qué se aprende:

Administración de Servidores (DBA), control de accesos (DCL - Data Control Language) y seguridad informática.
